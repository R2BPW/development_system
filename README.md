# Development System

Система для работы с вычислительными потоками через Telegram-бот. Мастер управляет потоками, Подмастерье их исполняет. Потоки порождаются через LLM и загружаются в живой образ без перезапуска.

## Архитектура

```
Telegram
   │
   ▼
Мастер (SBCL)          ←──── управление ────
   │                                        │
   │ записывает активный.scm                │
   ▼                                        │
Подмастерье (SBCL)                    Потоки (*.lisp)
   │                                        │
   │ load → funcall выполнить              │
   └────────────────────────────────────────┘
```

**Мастер** — Telegram-бот. Принимает команды, управляет потоками, хранит историю диалога.

**Подмастерье** — исполнитель. Держит живой SBCL-образ, загружает потоки по запросу, вызывает `выполнить/1` напрямую.

**Потоки** — файлы `*.lisp` в `мастер/потоки/`. Контракт: один пакет `:поток-ИМЯ`, одна функция `выполнить (задача) → строка`.

## Команды бота

| Команда | Описание |
|---|---|
| `/старт` | Показать меню |
| `/породить <описание>` | LLM генерирует новый поток, компилирует и загружает |
| `/запустить <имя> <задача>` | Выполнить задачу через указанный поток |
| `/потоки` | Список загруженных потоков |
| `/диалог <текст>` | Диалог с LLM с историей |
| `/сбросить` | Сбросить историю диалога |
| Свободный текст | Перенаправляется в `/диалог` |

## Встроенные потоки

| Поток | Описание |
|---|---|
| `эхо` | Возвращает задачу как есть (для отладки) |
| `уточнение` | Уточняет задачу через LLM |
| `исполнитель` | Выполняет задачу через LLM |
| `питон` | Генерирует и выполняет Python-код |
| `выполни-код` | LLM → Python-код → запуск через `uiop:run-program` → результат |
| `реакт` | ReAct-агент: рассуждение + вычисления |
| `порождатель` | Метапоток: порождает новые потоки через LLM |

## Запуск

### Требования

- SBCL + Quicklisp
- Python 3 (для потоков с кодом)
- Переменные окружения:

```bash
export MASTER_BOT_TOKEN="8005179888:AAH..."
export ADMIN_CHAT_ID="250203350"
export OPENROUTER_API_KEY="sk-or-v1-..."
```

### Мастер

```bash
sbcl --load ~/quicklisp/setup.lisp \
     --eval '(push #p"мастер/cl/" asdf:*central-registry*)' \
     --eval '(asdf:load-system :мастер :verbose nil)' \
     --eval '(мастер:start)'
```

### Подмастерье

```bash
cd подмастерье
sbcl --load ~/quicklisp/setup.lisp \
     --eval '(load "main.lisp")' \
     --eval '(подмастерье:start)'
```

## Ключевые решения

**Файл как IPC** — `мастер/потоки/активный.scm` содержит S-выражение `(активный (файл ...) (пакет ...))`. Подмастерье читает его при каждом сообщении. Нет сокетов, нет RPC.

**Персистентный образ** — потоки загружаются через `load` в живой SBCL-образ. Второй вызов того же потока — прямой `funcall`, без IO.

**Явное состояние** — реестр загруженных потоков и offset Telegram передаются как параметры рекурсии, не хранятся в глобальных переменных.

**Контракт потока** — одна функция `(выполнить задача)`. Обнаружение через `find-package` + `find-symbol` + `fboundp`.

**Метациркулярность** — поток `порождатель` генерирует новые потоки через LLM, компилирует их (`compile-file`) и загружает в тот же образ. Новый поток доступен немедленно.

## Структура

```
мастер/
  cl/              — Common Lisp исходники мастера
    мастер.asd
    packages.lisp
    config.lisp
    telegram.lisp  — Telegram API клиент
    llm.lisp       — OpenRouter LLM клиент
    dusha.lisp     — «душа» бота (личность)
    история.lisp   — история диалога
    потоки.lisp    — управление потоками
    команды.lisp   — обработчики команд
    main.lisp      — polling loop
  потоки/          — файлы потоков (*.lisp)
    порождатель.lisp
    выполни-код.lisp
    уточнение.lisp
    ...
подмастерье/       — Common Lisp исполнитель (без изменений)
docs/              — статьи (LaTeX)
```
