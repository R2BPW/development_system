# План рефакторинга development_system

## Диагноз — почему система неработоспособна

| # | Проблема | Где |
|---|----------|-----|
| 1 | Формат `душа.scm` не совпадает с `читать-душу` → системный промпт пустой | dusha.lisp |
| 2 | `%обр-старт` вызывает `send-message` сам, возвращает `nil` → нарушен контракт | команды.lisp |
| 3 | Кнопки «Запустить» → `/запустить` без аргументов → «Использование:…» | команды.lisp |
| 4 | LLM-запросы без таймаута блокируют polling loop | llm.lisp |
| 5 | `*llm-key*` вычисляется при загрузке → nil если ключ задан после | llm.lisp |
| 6 | История диалога без ограничения длины → LLM error | история.lisp |
| 7 | `загрузить-все-потоки` не заполняет `*активные-потоки*` → `/состояние` пустое | потоки.lisp |
| 8 | Тесты без моков требуют живые токены → CI невозможен | тесты.lisp |
| 9 | Смешанные HTTP: dexador в мастере, drakma в потоках | — |

## Фаза 1 — Запускаемость

### 1.1 Исправить `душа.scm` ↔ `читать-душу`
Привести формат файла к ожидаемому `читать-душу` или наоборот. Добавить проверку при старте с явной ошибкой.

### 1.2 Единый контракт обработчиков
Все `%обр-*` возвращают `(values строка &optional reply-markup)`. IO только в `%dispatch`:
```lisp
(multiple-value-bind (текст markup) (funcall fn chat-id args)
  (when (stringp текст) (send-message chat-id текст :reply-markup markup)))
```

### 1.3 Таймаут LLM
`dexador:*read-timeout*` = 60 сек. Обернуть в `handler-case` → «[таймаут LLM]».

### 1.4 Ленивое чтение ключа
```lisp
(defun %llm-key () (or *llm-key* (setf *llm-key* (uiop:getenv "OPENROUTER_API_KEY"))))
```

**Тесты фазы 1** (`тесты/unit.lisp`):
- `душа->системный-промпт` на fixture plist → содержит имя и стиль
- `%обр-старт` с моком → возвращает строку, не nil

## Фаза 2 — Инъекция зависимостей (моки)

### 2.1 Подменяемые функции
```lisp
;; telegram.lisp
(defvar *api-post-fn* #'%api-post-real)
(defun api-post (method payload) (funcall *api-post-fn* method payload))

;; llm.lisp
(defvar *llm-fn* #'%llm-real)
(defun llm-complete (prompt &rest args) (apply *llm-fn* prompt args))
```

### 2.2 Макрос для тестов
```lisp
(defmacro с-моками ((&key (api (constantly nil)) (llm (constantly "[мок]"))) &body body)
  `(let ((*api-post-fn* ,api) (*llm-fn* ,llm)) ,@body))
```

**Тесты фазы 2** (`тесты/unit.lisp`):
- `обработать-команду` с заглушкой IO → роутинг без side effects
- Свободный текст → LLM-mock вызван

## Фаза 3 — Юнит-тесты (без IO)

**`тесты/unit.lisp`** — SBCL + assert, запускается в CI без токенов:

| Функция | Проверка |
|---------|----------|
| `%split-words` | пустая строка → nil; "a  b" → ("a" "b") |
| `%cmd-args` | "/запустить эхо задача" → cmd + args |
| `%кнопка->команда` | точный матч кнопок |
| `%pkg-name` | "эхо" → "ПОТОК-ЭХО" |
| `имя-пакета` | "(defpackage :поток-питон ...)" → "питон" |
| `извлечь-из-блока` | ` ```lisp ... ``` ` → только код |

## Фаза 4 — Исправить состояние потоков и историю

### 4.1 Регистрация потоков в хеше
```lisp
(defun загрузить-все-потоки ()
  (dolist (путь (directory (merge-pathnames "*.lisp" *каталог-потоков*)))
    (when (загрузить-поток путь)
      (setf (gethash (pathname-name путь) *активные-потоки*) t))))
```

### 4.2 Ограничение истории
```lisp
(defparameter *макс-история* 20)
(defun добавить-сообщение (chat-id role content)
  (let* ((история (or (загрузить-историю chat-id) '()))
         (новая (last (append история (list `((:role . ,role) (:content . ,content))))
                      *макс-история*)))
    (сохранить-историю chat-id новая) новая))
```

**Тесты фазы 4** (`тесты/unit.lisp`):
- `добавить-сообщение` × 25 → длина ≤ `*макс-история*`
- `/состояние` после `загрузить-все-потоки` → непустой список

## Фаза 5 — Юзабилити

### 5.1 Интерактивный `/запустить`
Если аргументов нет → inline-клавиатура со списком потоков. Выбор → ожидание задачи.

### 5.2 Унификация HTTP-клиента
Все потоки переписать на dexador. Убрать drakma.

### 5.3 Фильтр доступа
`%dispatch`: если `chat-id ≠ *admin-chat-id*` → «Доступ запрещён.»

**Тесты фазы 5** (`тесты/master.lisp`):
- `/запустить` без аргументов → usage или клавиатура
- Чужой chat-id → «Доступ запрещён»

## Фаза 7 — Тесты потоков (требуют API-ключ)

**`тесты/streams.sh`** — SBCL subprocess, ручной прогон:

| Поток | Проверка |
|-------|----------|
| `эхо` | `(выполнить "ping")` → "ping" |
| `уточнение` | непустая строка |
| `выполни-код` | `"напечатай 42"` → содержит "42" |
| `реакт` | `"2*3"` → содержит "6" |
| `порождатель` | файл *.lisp создан, компилируется |

## Фаза 8 — Рефакторинг кода

### 8.1 Обработка ошибок
Один паттерн: `handler-case ... (error (e) (log-error e) fallback)`. Убрать голые `ignore-errors`.

### 8.2 Логирование
`(log/info fmt &rest args)` и `(log/error ...)` — timestamp на stdout.

### 8.3 Дедупликация потоков
`питон.lisp` и `исполнитель.lisp` делают одно → оставить один. `выполни-код.lisp` = 3 строки поверх.

## Edge cases

| Сценарий | Проверка |
|----------|----------|
| Offset persistence | рестарт → старые сообщения не дублируются |
| Rate limit 429 | api-post → nil → poll продолжается |
| Broken stream | синтаксическая ошибка → лог "FAIL", остальные грузятся |
| Nil text guard | `(send-message 123 nil)` → ничего не отправляется |

---

## CI entrypoint

**`тесты/run-ci.sh`**: запускает unit + master (без API-ключей), код выхода = число ошибок.

## Порядок выполнения

```
Ф1 запускаемость → Ф2 моки → Ф3 юниты → Ф4 состояние → Ф5 юзабилити → Ф6 интеграция → Ф7 потоки → Ф8 рефакторинг → Ф9 smoke
```

Каждая фаза коммитируется отдельно. После Ф3 CI без токенов должен быть зелёным.
