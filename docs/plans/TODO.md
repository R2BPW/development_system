# Техническое задание

## Двухуровневая система ботов для порождения и исполнения вычислительных потоков

*версия 1.0 — март 2026*

---

## 1. Замысел

Система состоит из двух самостоятельных Telegram-ботов и общей файловой среды между ними. Верхний уровень (оркестратор) порождает и управляет нижним (исполнитель). Оба уровня работают на одном Linux-сервере.

Цель — максимизировать удовольствие от работы с вычислительной машиной, придерживаясь принципов краткости, изящества и читаемости в духе SICP.

### 1.1. Участники системы

| Имя | Язык | Роль |
|-----|------|------|
| Мастер (бот-оркестратор) | Racket | Обсуждает задачи, порождает и запускает вычислительные потоки, хранит «душу» и историю |
| Подмастерье (бот-исполнитель) | Common Lisp (SBCL) | Принимает задачу от пользователя, прогоняет через текущий поток, возвращает ответ с гарантией доставки |
| ralphex (инструмент разработки) | Go (внешний) | Автономно выполняет планы разработки, проверяет код обоих ботов — не часть конечной системы |

### 1.2. Девиз (motto)

> Краткость. Изящные однострочники. Без англицизмов. В духе SICP.

## 2. Мастер — бот-оркестратор (Racket)

### 2.1. Общее описание

Постоянно запущенный процесс. Принимает сообщения через Telegram Bot API. Взаимодействует с Claude Code как с подпроцессом через командную строку. Результат работы — файл на диске с вычислительным потоком на Common Lisp, который может быть запущен в SBCL.

### 2.2. «Душа» — конфигурация

Хранится как S-выражение на диске. Содержит:

- Системное указание (system prompt) — описание личности и стиля
- Модель и температура (для передачи в Claude Code)
- Долговременная память — список фактов о пользователе (предпочтения, прошлые решения, контекст)
- Девиз (motto) — неизменяемая строка, управляющая стилем порождаемого кода

```scheme
(душа
  (указание  "Ты помощник для создания вычислительных потоков...")
  (модель    "claude-sonnet-4-20250514")
  (теплота   0.3)
  (память    ((предпочитает . "рекурсию а не циклы")
              (работает-над . "система порождения потоков")))
  (девиз     "краткость, однострочники, без англицизмов, SICP"))
```

### 2.3. Возможности

#### 2.3.1. Обсуждение задачи

Пользователь описывает задачу в свободной форме. Мастер ведёт диалог, уточняет, предлагает подход. Для глубокого анализа вызывает Claude Code как подпроцесс:

```scheme
(определить (спросить-клод вопрос)
  (пусть ((ответ (выполнить-процесс "claude" "--print" вопрос)))
    (разобрать-ответ ответ)))
```

#### 2.3.2. Порождение вычислительного потока

По результатам обсуждения Мастер формирует задание для Claude Code, получает код на Common Lisp и записывает его в файл. Поток оформляется как самодостаточная программа с точкой входа, обработкой условий и сохранением/восстановлением состояния.

#### 2.3.3. Запуск и остановка потока

Мастер может:

- Запустить SBCL-процесс с указанным файлом потока
- Отслеживать состояние процесса (работает / завершён / упал)
- Остановить процесс по команде пользователя
- Перезапустить из сохранённого образа (restore) при падении

#### 2.3.4. Повторное использование и комбинирование

Потоки хранятся в каталоге с именами и описаниями. Мастер умеет:

- Показать список существующих потоков
- Предложить повторно использовать существующий поток для новой задачи
- Комбинировать потоки — создать новый поток, составленный из вызовов существующих (последовательно, параллельно, по условию)
- Придумать новый способ комбинирования, если стандартные не подходят

#### 2.3.5. Показ времени и состояния

- `/время` — текущее время сервера
- `/состояние` — запущен ли поток, его имя, время работы
- `/душа` — показать текущую конфигурацию
- `/память` — показать и редактировать долговременную память
- `/потоки` — список сохранённых потоков

### 2.4. Структура каталогов

```
~/мастер/
  main.rkt              ; точка входа
  душа.scm              ; конфигурация «души»
  история/              ; журнал диалогов (по дням)
  потоки/               ; порождённые вычислительные потоки
    имя-потока.lisp     ; код потока
    имя-потока.мета     ; S-выражение с описанием
  образы/               ; SBCL-образы для восстановления
```

### 2.5. Стиль кода Racket

Идиоматический Racket в минималистичном духе:

- Маленькие чистые функции
- Структуры через `struct`, не через хеш-таблицы
- Сопоставление с образцом (`match`) вместо длинных условий
- Русские имена для прикладных функций, английские для системных
- Каждая функция умещается на одном экране (до 25 строк)

## 3. Подмастерье — бот-исполнитель (Common Lisp)

### 3.1. Общее описание

Отдельный Telegram-бот со своим @username. Запускается и останавливается Мастером. Внутри содержит один вычислительный поток (загруженный из файла). Общается с пользователем в режиме «задача → ответ».

### 3.2. Цикл работы

1. Пользователь пишет Подмастерью
2. Подмастерье спрашивает: «Какая задача?» (если не получил задачу сразу)
3. Загружает задачу в текущий вычислительный поток
4. Поток выполняет работу (возможно с несколькими вызовами к OpenRouter)
5. Подмастерье возвращает результат пользователю

### 3.3. Гарантия доставки ответа

Используются средства SBCL для обеспечения надёжности:

#### 3.3.1. Система условий и перезапуска

Каждый вычислительный поток оборачивается в `handler-bind` / `restart-case`. При ошибке (сбой сети, таймаут, некорректный ответ от языковой модели) — выбирается стратегия перезапуска:

- **повторить** — повторить вызов
- **пропустить** — вернуть частичный результат
- **подменить** — использовать запасное значение

```lisp
(restart-case
    (handler-bind
        ((ошибка-сети #'обработать-сеть))
      (выполнить-поток задача))
  (повторить () (выполнить-поток задача))
  (пропустить () (частичный-ответ))
  (подменить (знач) знач))
```

#### 3.3.2. Сохранение образа

Перед долгими вычислениями поток может сохранить образ через `save-lisp-and-die`. При падении Мастер перезапускает SBCL из сохранённого образа, и вычисление продолжается с последней точки сохранения.

### 3.4. Взаимодействие с OpenRouter

Вычислительные потоки обращаются к языковым моделям через OpenRouter. Минимальная обёртка:

- HTTP-запрос через Dexador или ручной сокет
- Формирование JSON через cl-json
- Разбор ответа с проверкой кода состояния
- Повторная попытка при временных ошибках (429, 502, 503)

### 3.5. Стиль кода Common Lisp

Минимализм в духе Scheme / SICP:

- Рекурсия вместо `loop` / `iterate`
- `defun` + `let` + `cond` + `funcall` — основной словарь
- Без CLOS (если только не нужна диспетчеризация по типу)
- Маленькие функции (до 15 строк)
- Русские имена для прикладных понятий

## 4. Взаимодействие компонентов

### 4.1. Схема потоков данных

```
Пользователь
    │
    ├──► [Telegram] ──► Мастер (Racket)
    │                      │
    │                      ├── Claude Code (подпроцесс)
    │                      │     └── порождает .lisp
    │                      │
    │                      └── управляет SBCL-процессом
    │                            └── Подмастерье
    │
    └──► [Telegram] ──► Подмастерье (SBCL)
                          │
                          └── OpenRouter (языковые модели)
```

### 4.2. Файловая среда — общий каталог

Оба процесса работают с общим каталогом `~/мастер/потоки/`. Мастер записывает файлы, SBCL читает. Синхронизация через файловую систему — никакого IPC сверх запуска процесса и чтения файлов.

### 4.3. Жизненный цикл потока

| Этап | Действующее лицо | Описание |
|------|-------------------|----------|
| 1. Замысел | Пользователь + Мастер | Обсуждение задачи в диалоге |
| 2. Порождение | Мастер + Claude Code | Создание .lisp файла потока |
| 3. Проверка | Мастер | Загрузка в SBCL, компиляция без выполнения |
| 4. Запуск | Мастер | Запуск SBCL-процесса с потоком, привязка к Подмастерью |
| 5. Работа | Подмастерье + OpenRouter | Приём задач, выполнение, возврат ответов |
| 6. Остановка | Мастер (по команде) | Завершение SBCL-процесса |

## 5. Формат вычислительного потока

### 5.1. Структура файла .lisp

Каждый поток — самодостаточный файл со следующими разделами:

```lisp
;;; -*- Mode: Lisp; Syntax: Common-Lisp -*-
;;; Имя: краткое-пересказ-задачи
;;; Создан: 2026-03-13
;;; Описание: ...

(defpackage :поток-имя (:use :cl))
(in-package :поток-имя)

;;; --- настройки ---
(defparameter *ключ-openrouter* (uiop:getenv "OPENROUTER_KEY"))
(defparameter *модель* "anthropic/claude-sonnet-4")

;;; --- ядро ---
(defun выполнить (задача)
  "Точка входа. Принимает строку задачи, возвращает ответ."
  (restart-case
      (обработать задача)
    (повторить () (выполнить задача))))

;;; --- взаимодействие с моделью ---
(defun спросить-модель (указание запрос)
  ...)

;;; --- точка сохранения ---
(defun сохранить-образ ()
  (sb-ext:save-lisp-and-die
    (merge-pathnames "образы/поток.core" (user-homedir-pathname))))
```

### 5.2. Файл метаданных .мета

```scheme
(мета
  (имя         "краткое-пересказ-задачи")
  (создан      "2026-03-13T14:30:00")
  (описание    "Что делает этот поток")
  (зависимости ())
  (входит-в    ())  ; список потоков, которые используют этот
  (состояние   :готов))  ; :готов | :работает | :ошибка
```

## 6. Ограничения и допущения

- Один пользователь — система рассчитана на единственного владельца
- Один активный поток — в каждый момент Подмастерье обслуживает один поток
- Linux — SBCL-образы привязаны к платформе, переносимость не требуется
- Локальная сеть или VPS — оба бота и Claude Code работают на одной машине
- Ключ OpenRouter — передаётся через переменную окружения
- Ключ Telegram Bot API — по одному на каждого бота, через переменные окружения

## 8. Критерии приёмки

1. Мастер отвечает в Telegram, показывает время и состояние
2. Мастер порождает синтаксически корректный `.lisp` через Claude Code
3. Порождённый `.lisp` загружается в SBCL без ошибок
4. Подмастерье принимает задачу и возвращает ответ через OpenRouter
5. При сбое сети поток перезапускается через `restart`, ответ доставляется
6. Образ сохраняется и восстанавливается корректно
7. Код Racket — идиоматичный, функции до 25 строк, читается целиком
8. Код CL — в духе SICP, функции до 15 строк, без CLOS и `loop`
9. Все планы выполняются через ralphex автономно
