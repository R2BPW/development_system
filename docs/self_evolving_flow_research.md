# Саморазвивающийся поток: архитектура обучающейся системы в development_system

## Аннотация

Данное исследование описывает архитектуру саморазвивающегося потока для системы `development_system` — метациркулярной среды вычислительных потоков на Common Lisp. Предлагается система из четырёх взаимосвязанных потоков: `память`, `обучение`, `диагностика` и `автоматор`, которые позволяют накапливать опыт решения задач, распознавать паттерны и автоматически порождать специализированные потоки. Исследование охватывает теоретические основы (in-context learning, episodic memory, self-improvement без fine-tuning), конкретную архитектуру в рамках существующей системы, примеры кода на Common Lisp и пошаговый план внедрения.

---

## 1. Введение

### 1.1. Постановка задачи

Артемий — разработчик на Python/Django/ГИС, который ежедневно решает типовые и нетиповые задачи. Многие решения повторяются: оптимизация N+1 запросов в GeoDjango, обработка геометрий, работа с PostGIS. Идея: создать систему, которая:

1. **Запоминает** пары (проблема → решение) из реальной практики
2. **Распознаёт** похожие проблемы и предлагает решения
3. **Автоматизирует** — при высокой уверенности порождает специализированный поток

### 1.2. Контекст системы

`development_system` — это:
- **Мастер** (SBCL-процесс) — принимает команды из Telegram
- **Поток** — файл `*.lisp` с пакетом `:поток-ИМЯ` и функцией `выполнить`
- **Порождатель** — метапоток, генерирующий новые потоки через LLM

Ключевое свойство: **метациркулярность** — слой генерации кода = слой исполнения кода. Новый поток компилируется и загружается в живой образ без перезапуска.

### 1.3. Цель исследования

Спроектировать архитектуру саморазвивающегося потока, который переходит от **имитации** (воспроизведение решений) к **автоматизации** (самостоятельное порождение специализированных потоков).

## 2. Теоретические основы

### 2.1. Few-shot и In-Context Learning

**In-context learning (ICL)** — способность LLM решать задачи на основе примеров в промпте без изменения весов модели. Механизм работы:

```
Промпт:
  Пример 1: проблема → решение
  Пример 2: проблема → решение
  Новая проблема: ???
  
LLM выводит решение по аналогии.
```

**Ключевые свойства ICL:**
- Работает без обучения — используем готовую модель
- Качество растёт с количеством релевантных примеров (до ~10-15)
- Порядок примеров влияет на результат — ближайшие к запросу должны быть последними
- Чем конкретнее формат примеров, тем стабильнее вывод

**Применение в нашей системе:** храним библиотеку пар (проблема → решение), при новой проблеме находим k ближайших и включаем в промпт.

### 2.2. Fine-tuning vs In-Context Learning

| Характеристика | Fine-tuning | ICL |
|----------------|-------------|-----|
| Изменение весов | Да | Нет |
| Стоимость | Высокая (обучение) | Низкая (только токены) |
| Скорость адаптации | Часы/дни | Мгновенно |
| Забывание | Да (catastrophic forgetting) | Нет |
| Гибкость | Низкая (фиксированная модель) | Высокая (динамические примеры) |

**Вывод:** для нашей задачи **fine-tuning не нужен**. ICL даёт достаточное качество при минимальных затратах, позволяет мгновенно добавлять новые примеры и не требует инфраструктуры обучения.

### 2.3. Episodic Memory для агентов

**Episodic memory** — память о конкретных эпизодах (событиях), в отличие от semantic memory (общие знания).

#### MemGPT (2023)

Идея: LLM как операционная система с иерархической памятью:
- **Main context** — рабочая память (окно контекста)
- **Archival memory** — долговременное хранилище с поиском
- **Recall memory** — поиск по ключевым словам

Применимо к нашей системе: `поток-память` = archival memory с поиском по сходству.

#### RAG over Examples

Retrieval-Augmented Generation применительно к примерам:
1. Encode проблему → embedding
2. Поиск k ближайших проблем в хранилище
3. Добавление примеров в промпт
4. Генерация решения

**Проблема:** нужен embedding-сервис или локальная модель. Альтернатива — поиск через сам LLM (дороже, но проще).

### 2.4. Self-Improvement без обучения модели

#### RLHF и DPO — не применимо

RLHF (Reinforcement Learning from Human Feedback) и DPO (Direct Preference Optimization) требуют:
- Доступа к весам модели
- Инфраструктуры обучения
- Большого количества данных

Для нашей задачи **избыточно**.

#### Constitutional AI — частично применимо

Идея: модель сама оценивает свои ответы по набору принципов.

**Применение:** `поток-диагностика` может включать self-critique:
```
Вот решение: ...
Оцени по критериям:
1. Решает ли проблему?
2. Есть ли побочные эффекты?
3. Насколько похоже на известные решения?
```

#### Prompt-based Self-Improvement

Практический подход для нашей системы:
1. Генерируем решение
2. Просим LLM оценить уверенность (0-100)
3. Если уверенность > порога, применяем автоматически
4. Если уверенность низкая, просим уточнение у человека

### 3.2. Поток `память`

**Назначение:** долговременное хранилище пар (проблема → решение) с поиском по сходству.

**Интерфейс:**
```lisp
(выполнить "добавить: проблема=N+1 в GeoDjango | решение=prefetch_related")
(выполнить "найти: медленный queryset с Point")
(выполнить "список")
(выполнить "удалить: 3")  ; по индексу
```

**Формат хранения:** JSON-файл в простой структуре:

```json
{
  "версия": 1,
  "примеры": [
    {
      "id": 1,
      "проблема": "N+1 запросы в GeoDjango при обходе связанных объектов с геометрией",
      "решение": "select_related для FK, prefetch_related для M2M, defer('geom') если геометрия не нужна",
      "теги": ["django", "geodjango", "n+1", "оптимизация"],
      "дата": "2026-03-14",
      "оценка": 1,
      "применений": 3
    }
  ]
}
```

**Поиск по сходству** — три стратегии (по возрастанию качества/стоимости):

1. **Keyword matching:** разбить проблему на слова, искать пересечение
2. **LLM-сравнение:** запрос "Похожа ли проблема A на проблему B? Ответь да/нет/частично"
3. **Embedding + cosine similarity:** использовать OpenRouter/OpenAI embedding API

Рекомендация для MVP: **LLM-сравнение** — простая реализация, достаточное качество.

### 3.3. Поток `обучение`

**Назначение:** принимает новую пару от Артемия, валидирует, сохраняет в память, анализирует паттерн.

**Интерфейс:**
```lisp
(выполнить "проблема: N+1 в GeoDjango queryset
решение: select_related('region') + prefetch_related('points')
теги: django, geodjango, n+1")
```

**Логика:**
1. Парсинг входа (проблема, решение, теги)
2. Проверка на дубликаты через `поток-память`
3. Если похожая проблема уже есть — предложить обновить или добавить как вариант
4. Сохранение через `поток-память`
5. Анализ паттерна: "Это новый паттерн или вариация существующего?"

### 3.4. Поток `диагностика`

**Назначение:** при новой проблеме ищет похожие, оценивает применимость, предлагает решение.

**Интерфейс:**
```lisp
(выполнить "медленный queryset с Point полями в GeoDjango")
```

**Логика:**
1. Запросить `поток-память` для поиска похожих проблем
2. Если найдены — отправить в LLM с контекстом:
   ```
   Найдены похожие проблемы и решения:
   1. [проблема] → [решение]
   2. ...
   
   Новая проблема: [текст]
   
   Предложи решение, основываясь на похожих случаях.
   Оцени уверенность (0-100).
   ```
3. Вернуть предложение с уверенностью
4. Обновить счётчик применений в памяти

### 3.5. Поток `автоматор`

**Назначение:** если уверенность высокая и паттерн повторяется, порождает специализированный поток.

**Триггеры автоматизации:**
- Уверенность > 85%
- Паттерн встречался > 3 раз
- Пользователь подтверждал решение > 2 раз

**Логика:**
1. Проверить триггеры
2. Сформировать описание для `порождатель`:
   ```
   Создай поток оптимизатор-гео-запросов.
   Задача: принять описание медленного GeoDjango queryset и вернуть оптимизированный код.
   Известные паттерны:
   - N+1 → select_related + prefetch_related
   - Лишняя геометрия → defer('geom')
   - ...
   ```
3. Вызвать `поток-порождатель`
4. Уведомить пользователя

### 3.6. Определение сходства проблем

**Стратегия LLM-сравнения (рекомендуемая для MVP):**

```lisp
(defun похожи-p (проблема-1 проблема-2)
  "Возвращает :да, :частично или :нет"
  (let* ((промпт (format nil "Сравни две проблемы программиста:
Проблема А: ~A
Проблема Б: ~A

Они похожи? Ответь ОДНИМ словом: да, частично, нет" проблема-1 проблема-2))
         (ответ (string-downcase (string-trim '(#\Space #\Newline) (llm-complete промпт)))))
    (cond ((search "да" ответ) :да)
          ((search "частично" ответ) :частично)
          (t :нет))))
```

**Стратегия поиска k ближайших:**

```lisp
(defun найти-похожие (новая-проблема примеры &key (k 3))
  "Возвращает k наиболее похожих примеров."
  (let* ((с-оценками (mapcar (lambda (пример)
                               (cons (похожи-p новая-проблема (cdr (assoc :проблема пример)))
                                     пример))
                             примеры))
         (отсортированы (sort с-оценками
                              (lambda (a b)
                                (> (case (car a) (:да 2) (:частично 1) (:нет 0))
                                   (case (car b) (:да 2) (:частично 1) (:нет 0)))))))
    (mapcar #'cdr (subseq отсортированы 0 (min k (length отсортированы))))))
```

### 3.7. Порог уверенности

**Структура порога:**

```lisp
(defparameter *пороги*
  '((:предложение . 30)      ; минимум для вывода предложения
    (:применение . 70)       ; автоматическое применение с подтверждением
    (:автоматизация . 85)))  ; порождение нового потока
```

**Динамическая настройка:**
- После каждого feedback обновляем статистику
- Если точность падает ниже 70%, повышаем порог на 5%
- Если точность выше 90%, понижаем порог на 5%

## 5. Реализация

### 5.1. Поток `память` (поток-память.lisp)

```lisp
;;;; поток-память.lisp
;;;; Долговременная память для пар проблема→решение

(defpackage :поток-память
  (:use :cl)
  (:export #:выполнить
           #:добавить-пример
           #:найти-похожие
           #:получить-все
           #:удалить-пример
           #:обновить-оценку))

(in-package :поток-память)

;;; === Конфигурация ===

(defparameter *путь-памяти*
  (merge-pathnames "память.json"
                   (or (uiop:getenv "MEMORY_DIR")
                       (user-homedir-pathname))))

(defparameter *модель* "openai/gpt-4.1")

;;; === Работа с хранилищем ===

(defun загрузить-память ()
  "Загружает JSON из файла, возвращает alist."
  (if (probe-file *путь-памяти*)
      (with-open-file (f *путь-памяти* :external-format :utf-8)
        (cl-json:decode-json f))
      '((:версия . 1) (:примеры . nil) (:паттерны . nil))))

(defun сохранить-память (данные)
  "Сохраняет alist в JSON."
  (ensure-directories-exist *путь-памяти*)
  (with-open-file (f *путь-памяти*
                     :direction :output
                     :if-exists :supersede
                     :external-format :utf-8)
    (cl-json:encode-json данные f))
  t)

(defun примеры ()
  "Возвращает список примеров."
  (cdr (assoc :примеры (загрузить-память))))

(defun следующий-id ()
  "Генерирует следующий ID."
  (1+ (reduce #'max (mapcar (lambda (p) (or (cdr (assoc :id p)) 0))
                            (примеры))
              :initial-value 0)))

;;; === Сходство через LLM ===

(defun ключ-апи ()
  (or (uiop:getenv "OPENROUTER_API_KEY")
      (error "OPENROUTER_API_KEY не задан")))

(defun llm-запрос (промпт)
  "Простой запрос к LLM."
  (let* ((тело (cl-json:encode-json-to-string
                `((:model . ,*модель*)
                  (:messages . (((:role . "user") (:content . ,промпт)))))))
         (сырой (dexador:post "https://openrouter.ai/api/v1/chat/completions"
                              :headers `(("Authorization" . ,(format nil "Bearer ~A" (ключ-апи)))
                                         ("Content-Type" . "application/json"))
                              :content тело
                              :read-timeout 30))
         (стр (if (stringp сырой) сырой (sb-ext:octets-to-string сырой :external-format :utf-8)))
         (json (cl-json:decode-json-from-string стр)))
    (cdr (assoc :content (cdr (assoc :message (first (cdr (assoc :choices json)))))))))

(defun похожи-p (проблема-1 проблема-2)
  "Сравнивает две проблемы, возвращает :да, :частично, :нет"
  (let* ((промпт (format nil "Сравни две проблемы разработчика.~%~
                              Проблема А: ~A~%~
                              Проблема Б: ~A~%~%~
                              Похожи ли они по сути (одинаковый тип проблемы, ~
                              схожее решение подойдёт)?~%~
                              Ответь ОДНИМ словом: да, частично, нет"
                         проблема-1 проблема-2))
         (ответ (string-downcase (string-trim '(#\Space #\Newline #\Return) (llm-запрос промпт)))))
    (cond ((search "да" ответ) :да)
          ((search "частично" ответ) :частично)
          (t :нет))))

(defun оценка-сходства (сходство)
  (case сходство (:да 2) (:частично 1) (t 0)))

;;; === Публичный API ===

(defun добавить-пример (проблема решение &key теги)
  "Добавляет новый пример в память."
  (let* ((память (загрузить-память))
         (примеры-текущие (cdr (assoc :примеры память)))
         (новый `((:id . ,(следующий-id))
                  (:проблема . ,проблема)
                  (:решение . ,решение)
                  (:теги . ,теги)
                  (:дата . ,(multiple-value-bind (s m h d mo y) (decode-universal-time (get-universal-time))
                              (declare (ignore s m h))
                              (format nil "~4,'0D-~2,'0D-~2,'0D" y mo d)))
                  (:оценка . 0)
                  (:применений . 0)))
         (обновлённая (cons новый примеры-текущие)))
    (setf (cdr (assoc :примеры память)) обновлённая)
    (сохранить-память память)
    (cdr (assoc :id новый))))

(defun найти-похожие (проблема &key (k 3))
  "Находит k наиболее похожих примеров."
  (let* ((все (примеры))
         (с-оценками (mapcar (lambda (п)
                               (cons (похожи-p проблема (cdr (assoc :проблема п))) п))
                             все))
         (отсортированы (sort с-оценками
                              (lambda (a b) (> (оценка-сходства (car a))
                                               (оценка-сходства (car b)))))))
    (remove-if (lambda (x) (eq (car x) :нет))
               (subseq отсортированы 0 (min k (length отсортированы))))))

(defun получить-все ()
  "Возвращает все примеры."
  (примеры))

(defun удалить-пример (id)
  "Удаляет пример по ID."
  (let* ((память (загрузить-память))
         (примеры-текущие (cdr (assoc :примеры память)))
         (без-удалённого (remove-if (lambda (п) (= (cdr (assoc :id п)) id))
                                    примеры-текущие)))
    (setf (cdr (assoc :примеры память)) без-удалённого)
    (сохранить-память память)
    (not (= (length примеры-текущие) (length без-удалённого)))))

(defun обновить-оценку (id дельта)
  "Изменяет оценку примера на дельту (+1 хорошо, -1 плохо)."
  (let* ((память (загрузить-память))
         (примеры-текущие (cdr (assoc :примеры память)))
         (обновлённые (mapcar (lambda (п)
                                (if (= (cdr (assoc :id п)) id)
                                    (progn
                                      (setf (cdr (assoc :оценка п))
                                            (+ (or (cdr (assoc :оценка п)) 0) дельта))
                                      (setf (cdr (assoc :применений п))
                                            (1+ (or (cdr (assoc :применений п)) 0)))
                                      п)
                                    п))
                              примеры-текущие)))
    (setf (cdr (assoc :примеры память)) обновлённые)
    (сохранить-память память)))

;;; === Точка входа ===

(defun парсить-команду (текст)
  "Разбирает текст команды."
  (cond
    ((search "добавить:" текст)
     (let* ((после (subseq текст (+ (search "добавить:" текст) 9)))
            (поз-решения (search "решение=" после)))
       (if поз-решения
           (list :добавить
                 (string-trim '(#\Space) (subseq после 0 (search "|" после)))
                 (string-trim '(#\Space) (subseq после (+ poз-решения 8))))
           (list :ошибка "Формат: добавить: проблема=... | решение=..."))))
    ((search "найти:" текст)
     (list :найти (string-trim '(#\Space) (subseq текст (+ (search "найти:" текст) 6)))))
    ((search "список" текст)
     (list :список))
    ((search "удалить:" текст)
     (list :удалить (parse-integer (string-trim '(#\Space) (subseq текст (+ (search "удалить:" текст) 8))))))
    (t (list :найти текст))))  ; по умолчанию — поиск

(defun форматировать-пример (сходство-и-пример)
  "Форматирует пример для вывода."
  (let* ((сходство (car сходство-и-пример))
         (п (cdr сходство-и-пример))
         (id (cdr (assoc :id п)))
         (проблема (cdr (assoc :проблема п)))
         (решение (cdr (assoc :решение п)))
         (процент (case сходство (:да "90%") (:частично "60%") (t "30%"))))
    (format nil "[~A] (~A сходство)~%Проблема: ~A~%Решение: ~A~%"
            id процент проблема решение)))

(defun выполнить (задача)
  "Главная точка входа потока."
  (handler-case
      (let ((команда (парсить-команду задача)))
        (case (first команда)
          (:добавить
           (let ((id (добавить-пример (second команда) (third команда))))
             (format nil "✅ Сохранено (id=~A)" id)))
          (:найти
           (let ((найденные (найти-похожие (second команда))))
             (if найденные
                 (format nil "🔍 Найдено ~A похожих:~%~%~{~A~%~}"
                         (length найденные)
                         (mapcar #'форматировать-пример найденные))
                 "Похожих проблем не найдено.")))
          (:список
           (let ((все (получить-все)))
             (if все
                 (format nil "📚 Всего примеров: ~A~%~%~{- [~A] ~A~%~}"
                         (length все)
                         (mapcan (lambda (п)
                                   (list (cdr (assoc :id п))
                                         (cdr (assoc :проблема п))))
                                 все))
                 "Память пуста.")))
          (:удалить
           (if (удалить-пример (second команда))
               (format nil "🗑️ Пример ~A удалён" (second команда))
               "Пример не найден."))
          (:ошибка (second команда))
          (t "Неизвестная команда. Доступны: добавить, найти, список, удалить")))
    (error (e) (format nil "❌ Ошибка: ~A" e))))
```

```lisp
(defun парсить-ввод (текст)
  "Извлекает проблему, решение и теги из текста.
   Формат: проблема: ... решение: ... теги: ..."
  (let ((проблема nil)
        (решение nil)
        (теги nil))
    ;; Ищем маркеры
    (let ((поз-пробл (search "проблема:" (string-downcase текст)))
          (поз-реш (search "решение:" (string-downcase текст)))
          (поз-теги (search "теги:" (string-downcase текст))))
      (when (and поз-пробл поз-реш)
        (setf проблема (string-trim '(#\Space #\Newline)
                                    (subseq текст
                                            (+ поз-пробл 9)
                                            поз-реш)))
        (setf решение (string-trim '(#\Space #\Newline)
                                   (subseq текст
                                           (+ поз-реш 8)
                                           (or поз-теги (length текст)))))
        (when поз-теги
          (setf теги (mapcar (lambda (t) (string-trim '(#\Space) t))

                                                :separator ","))))))

                             проблема
                             (mapcar (lambda (п) (cdr (assoc :проблема (cdr п)))) похожие)))
             (анализ (llm-запрос промпт)))
        (if (search "вариация" (string-downcase анализ))

            (values :новый анализ)))))

      (multiple-value-bind (проблема решение теги) (парсить-ввод задача)

          (return-from выполнить
            "❌ Не удалось распознать. Формат:
проблема: <описание проблемы>
решение: <как решили>
теги: тег1, тег2 (опционально)"))

          ;; Проверяем на дубликат

                          Добавить как вариант? (пока не реализовано)"

            ;; Анализируем паттерн

                          ~A~%~
```

```lisp
;;;; поток-автоматор.lisp
;;;; Автоматическое порождение специализированных потоков при устойчивых паттернах

(defpackage :поток-автоматор
  (:use :cl)
  (:export #:выполнить
           #:проверить-паттерны
           #:породить-специализированный))

(unless (find-package :поток-порождатель)
  (load (merge-pathnames "порождатель.lisp" *load-pathname*) :verbose nil))

;;; === Пороги ===

(defparameter *мин-случаев* 3
  "Минимум случаев для рассмотрения автоматизации")

(defparameter *мин-успешность* 0.8
  "Минимальная доля успешных применений")

(defparameter *мин-уверенность* 85
  "Минимальная средняя уверенность")

;;; === Анализ паттернов ===

(defun группировать-по-паттерну (примеры)
  "Группирует примеры по сходству (простая эвристика по тегам)."
  (let ((группы (make-hash-table :test #'equal)))
    (dolist (п примеры)
      (let ((ключ (or (first (cdr (assoc :теги п))) "общее")))
        (push п (gethash ключ группы nil))))
    (loop for k being the hash-keys of группы
          using (hash-value v)
          collect (cons k v))))

(defun статистика-группы (группа)
  "Вычисляет статистику для группы примеров."
  (let* ((примеры (cdr группа))
         (кол-во (length примеры))
         (успешных (count-if (lambda (п) (> (or (cdr (assoc :оценка п)) 0) 0)) примеры))
         (применений (reduce #'+ (mapcar (lambda (п) (or (cdr (assoc :применений п)) 0)) примеры))))
    (list :название (car группа)
          :случаев кол-во
          :успешность (if (> кол-во 0) (/ успешных кол-во) 0)
          :применений применений
          :примеры примеры)))

(defun готов-к-автоматизации-p (статистика)
  "Проверяет, готов ли паттерн к автоматизации."
  (and (>= (getf статистика :случаев) *мин-случаев*)
       (>= (getf статистика :успешность) *мин-успешность*)
       (>= (getf статистика :применений) (* 2 (getf статистика :случаев)))))

;;; === Генерация описания для порождателя ===

(defun сформировать-описание (статистика)
  "Формирует описание для порождателя специализированного потока."
  (let* ((название (getf статистика :название))
         (примеры (getf статистика :примеры))
         (паттерны (format nil "~{- ~A → ~A~%~}"
                           (mapcan (lambda (п)
                                     (list (cdr (assoc :проблема п))
                                           (cdr (assoc :решение п))))
                                   (subseq примеры 0 (min 5 (length примеры)))))))
    (format nil "Создай поток ~A-оптимизатор.~%~
                 Задача: принять описание проблемы в области ~A и вернуть решение.~%~%~
                 Известные паттерны проблема→решение:~%~A~%~
                 Поток должен анализировать входную проблему и применять ~
                 подходящий паттерн. Если ни один не подходит — вернуть ~
                 'Паттерн не распознан'."
            название название паттерны)))

(defun проверить-паттерны ()
  "Проверяет все паттерны и возвращает готовые к автоматизации."
  (let* ((примеры (поток-память:получить-все))
         (группы (группировать-по-паттерну примеры))
         (статистики (mapcar #'статистика-группы группы))
         (готовые (remove-if-not #'готов-к-автоматизации-p статистики)))
    готовые))

(defun породить-специализированный (статистика)
  "Порождает специализированный поток для паттерна."
  (let ((описание (сформировать-описание статистика)))
    (поток-порождатель:выполнить описание)))

(defun выполнить (задача)
  "Точка входа: проверяет паттерны и предлагает/выполняет автоматизацию."
  (handler-case
      (cond
        ;; Команда: породить конкретный паттерн
        ((search "породить:" (string-downcase задача))
         (let* ((название (string-trim '(#\Space)
                                       (subseq задача (+ (search "породить:" (string-downcase задача)) 9))))
                (готовые (проверить-паттерны))
                (целевой (find название готовые :key (lambda (s) (getf s :название)) :test #'string-equal)))
           (if целевой
               (породить-специализированный целевой)
               (format nil "Паттерн '~A' не найден или не готов к автоматизации" название))))
        
        ;; Команда: статус
        ((search "статус" (string-downcase задача))
         (let ((готовые (проверить-паттерны)))
           (if готовые
               (format nil "🤖 Паттерны, готовые к автоматизации:~%~%~{~A~%~}"
                       (mapcar (lambda (s)
                                 (format nil "• ~A (случаев: ~A, успешность: ~A%)~%  /автоматор породить: ~A"
                                         (getf s :название)
                                         (getf s :случаев)
                                         (round (* 100 (getf s :успешность)))
                                         (getf s :название)))
                               готовые))
               "Пока нет паттернов, готовых к автоматизации.
Нужно минимум 3 случая с хорошей успешностью.")))
        
        ;; По умолчанию: показать статус
        (t (выполнить "статус")))
    (error (e) (format nil "❌ Ошибка автоматора: ~A" e))))
```

## 6. Эволюция и обратная связь

### 6.1. Жизненный цикл потока обучения

```
ЭТАП 1: НАКОПЛЕНИЕ (< 3 примеров)
├── Пользователь добавляет пары проблема→решение
├── Система сохраняет в память
└── Статус: "Учусь, нужно больше примеров"

ЭТАП 2: ИМИТАЦИЯ (3-10 примеров)
├── При новой проблеме ищем похожие
├── Формируем промпт с примерами (few-shot)
├── Генерируем решение с уверенностью
└── Статус: "Могу предлагать, но проверяй"

ЭТАП 3: ОБОБЩЕНИЕ (10+ примеров, >70% успешность)
├── Выделяем паттерны из примеров
├── Формируем абстрактные правила
├── Повышаем уверенность
└── Статус: "Понимаю паттерн, уверен в решениях"

ЭТАП 4: АВТОМАТИЗАЦИЯ (>3 случаев, >80% успешность)
├── Автоматор предлагает породить специализированный поток
├── Пользователь подтверждает
├── Порождатель создаёт новый поток
└── Статус: "Порождён поток Х-оптимизатор"
```

### 6.2. Механизм обратной связи

```lisp
;;; Пользователь даёт feedback через команды:

;; После предложения решения:
/хорошо 5           ; id примера, который помог
/плохо 5 причина    ; id примера + почему не подошло

;;; Обработка в потоке:
(defun обработать-feedback (команда)
  (cond
    ((search "хорошо" команда)
     (let ((id (извлечь-id команда)))
       (поток-память:обновить-оценку id +1)
       "✅ Спасибо! Запомнил как хорошее решение."))
    
    ((search "плохо" команда)
     (let ((id (извлечь-id команда))
           (причина (извлечь-причину команда)))
       (поток-память:обновить-оценку id -1)
       ;; Сохраняем причину как негативный пример
       (поток-память:добавить-пример
        (format nil "НЕ ДЕЛАТЬ: ~A" (получить-проблему id))
        (format nil "Причина: ~A" причина)
        :теги '("антипаттерн"))
       "📝 Запомнил как плохое решение. Спасибо за уточнение."))))
```

### 6.3. Защита от плохих паттернов

**Проблема:** garbage in → garbage out. Если Артемий добавит неправильное решение, система будет его воспроизводить.

**Защитные механизмы:**

1. **Decay оценки со временем:**
```lisp
(defun применить-decay ()
  "Уменьшает влияние старых примеров без недавних применений."
  (dolist (п (поток-память:получить-все))
    (let* ((дней-с-применения (- (get-universal-time)
                                 (or (cdr (assoc :последнее-применение п))
                                     (cdr (assoc :дата п)))))
           (дней (/ дней-с-применения 86400)))
      (when (> дней 30)
        ;; Помечаем как устаревший
        (поток-память:обновить-оценку (cdr (assoc :id п)) -0.1)))))
```

2. **Порог применений:** пример должен быть применён минимум 2 раза с положительным feedback прежде чем влиять на автоматизацию.

3. **Self-critique:** перед автоматизацией запрашиваем у LLM оценку:
```lisp
(defun самокритика (паттерн)
  (llm-запрос 
   (format nil "Оцени этот паттерн решения проблем:~%~A~%~%~
               Возможные проблемы? Риски? Ограничения?~%~
               Ответь списком."
           паттерн)))
```

4. **Карантин для новых потоков:** автоматически порождённый поток первые 5 вызовов работает в режиме "предложение", не "применение".

### 6.4. Self-critique в диагностике

```lisp
(defun диагностика-с-критикой (проблема)
  "Двухэтапная диагностика: генерация + критика."
  (let* ((похожие (поток-память:найти-похожие проблема))
         (первичное-решение (сгенерировать-решение проблема похожие))
         (критика (llm-запрос
                   (format nil "Предложено решение:~%~A~%~%~
                               Проблема: ~A~%~%~
                               Критически оцени:~%~
                               1. Точно ли решает эту проблему?~%~
                               2. Возможные побочные эффекты?~%~
                               3. Нужны ли уточнения?~%~
                               Если нужна корректировка — предложи."
                           первичное-решение проблема)))
    (format nil "~A~%~%🔍 Самопроверка:~%~A" первичное-решение критика)))
```

## 8. Риски и ограничения

### 8.1. Контекстное окно LLM

**Проблема:** при накоплении сотен примеров невозможно включить все в промпт.

**Митигация:**
- Жёсткий лимит: топ-5 похожих примеров в промпте
- Агрегация: объединять похожие примеры в один
- Compression: использовать краткие формулировки

### 8.2. Drift качества

**Проблема:** со временем качество решений может деградировать из-за:
- Устаревших примеров
- Накопления ошибок
- Изменения контекста (новая версия Django, новые паттерны)

**Митигация:**
- Периодическая ревизия (ежемесячно)
- Автоматический decay
- Тегирование по версиям (`django-4.2`, `geodjango-2024`)

### 8.3. Overfitting на личный стиль

**Проблема:** система учится конкретному стилю Артемия, который может быть:
- Не оптимальным
- Не универсальным
- Основанным на устаревших практиках

**Митигация:**
- Периодическая валидация через внешние источники (документация, Stack Overflow)
- Флаг "проверено" для примеров, подтверждённых внешними источниками
- Self-critique с запросом "Соответствует ли это best practices?"

### 8.4. Безопасность порождённых потоков

**Проблема:** автоматически созданный код может содержать:
- Уязвимости
- Бесконечные циклы
- Опасные операции

**Митигация:**
1. **Sandboxing:** потоки не имеют доступа к файловой системе за пределами `/tmp`
2. **Timeout:** все вызовы ограничены по времени (60 сек)
3. **Code review prompt:** порождатель включает в промпт требования безопасности
4. **Карантин:** новые потоки первые N вызовов требуют подтверждения

### 8.5. Стоимость LLM-запросов

**Проблема:** каждый поиск по сходству = N запросов к LLM (где N = количество примеров).

**Митигация:**
- Batch-сравнение: один запрос "Какие из этих проблем похожи на X?"
- Кэширование результатов сравнения
- Предварительная фильтрация по ключевым словам
- Использование дешёвых моделей для сравнения (`gpt-4o-mini`)

**Оценка стоимости:**
- 100 примеров × batch-сравнение = ~500 токенов = ~$0.001
- Приемлемо для личного использования

## 9. Заключение

### 9.1. Ключевые выводы

1. **In-context learning достаточно** для задачи персонального обучения. Fine-tuning избыточен и создаёт больше проблем (catastrophic forgetting, инфраструктура), чем решает.

2. **Метациркулярность системы** — главное преимущество. Порождатель уже существует; саморазвивающийся поток — естественное расширение.

3. **Gradual autonomy** — правильная стратегия. Система не должна сразу автоматизировать; путь от имитации к автоматизации с явными checkpoint'ами (feedback, пороги, карантин).

4. **Human-in-the-loop** критичен. Обратная связь — не опция, а необходимость. Без неё система неизбежно деградирует.

### 9.2. Минимальный путь к работающей системе

```
День 1: память.lisp (keyword matching)
        ↓
День 2-3: обучение.lisp + диагностика.lisp (LLM)
        ↓
Неделя 1: feedback loop
        ↓
Неделя 2: автоматор.lisp
        ↓
Месяц 1: production-ready
```

### 9.3. Критерий успеха

Система успешна, если через месяц использования:
- Артемий тратит меньше времени на повторяющиеся проблемы
- Есть хотя бы 1 автоматически порождённый полезный поток
- Точность предложений > 70%
